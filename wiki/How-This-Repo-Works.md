# How This Repo Works

This page explains the full architecture of the ATL BIM Standards repo — intended for BIM Team members and anyone who wants to understand how the site is built, versioned, and deployed.

---

## The Five-Layer Stack

```
YOUR MARKDOWN FILES  (docs/*.md)
        ↓  rendered by
    MkDocs + Material Theme  (mkdocs.yml controls everything)
        ↓  versioned by
    Mike  (manages the gh-pages branch)
        ↓  deployed by
    GitHub Actions  (triggers on git tags like v1.0)
        ↓  served from
    bim.atlstandards.com  (FTP target)
```

---

## What Each Tool Does

**MkDocs** converts Markdown files into a static HTML website. `mkdocs serve` does this live in memory on your local machine. `mkdocs build` writes it to a `site/` folder.

**MkDocs Material** is the visual theme. It provides the layout, colors, search, navigation tabs, cards, admonitions, and the version selector dropdown.

**mkdocs-macros** lets you use Python variables inside Markdown. When a page contains `{{ standards['GN-001'].title }}`, the macros plugin looks that up in `main.py` and substitutes the value at build time. All standards metadata (version, date, title, description) lives in `main.py` — not in the individual `.md` files.

**GLightbox** is a plugin that makes every image click-to-expand in a full-screen overlay. No markdown changes needed.

**Mike** sits on top of MkDocs and manages versioned snapshots. Instead of building one site, Mike builds a snapshot and commits it to a dedicated `gh-pages` branch. Each version lives in its own subfolder:

```
gh-pages branch (what the world sees):
├── versions.json       ← Mike's index of all versions
├── 1.0/                ← frozen snapshot of v1.0, forever
├── 1.1/                ← frozen snapshot of v1.1
└── latest → 1.1        ← alias, always points to newest
```

The Material theme reads `versions.json` and shows the version selector dropdown in the site header.

**GitHub Actions** is the automation layer. The `deploy.yml` workflow triggers **only when a tag starting with `v` is pushed** (e.g. `v1.1`). It installs dependencies, generates the changelog, runs Mike to build and version the site, then FTP-deploys the `gh-pages` branch to `bim.atlstandards.com`. You never run Mike or FTP manually in production.

## Branch Strategy

| Branch | Purpose |
| --- | --- |
| `develop` | Active development — all edits, drafts, and fixes happen here |
| `main` | Stable, release-ready content. Merged from `develop` via PR. Version tags are created here. |
| `gh-pages` | Mike-managed versioned output. **Never edit manually.** |

---

## Release Workflow

```
1. Edit content on develop
2. Test locally with: mkdocs serve
3. Open PR on GitHub: develop → main
4. Review and merge the PR
5. Pull main locally:
       git checkout main
       git pull
6. Tag the release:
       git tag v1.1
       git push origin v1.1
7. Watch the Action run at github.com/Ktylos/ATL-BIM-STDS/actions
```

That's it. GitHub Actions handles everything after step 6.

## How a Consultant Accesses an Old Version

1. Their contract references ATL BIM Standards v1.0 (March 2026)
2. They visit `bim.atlstandards.com` — the site shows the latest version
3. They click the version selector dropdown → select `1.0`
4. The URL becomes `bim.atlstandards.com/1.0/`
5. That folder is a **frozen snapshot** — Mike never overwrites it, only adds new versions alongside it

Consultants should bookmark their version URL and note it in project records.

## Updating Standards Metadata

Version numbers, dates, and titles for all standards are defined in `main.py` — not in the individual `.md` files. To update the version across the entire site:

1. Open `main.py`
2. Find the relevant entry in `env.variables['standards']`
3. Update `version` and `date`
4. Save — the change propagates to every page that references that variable at build time

Never edit version numbers directly in `.md` files.

## The Changelog

The changelog at `docs/resources/changelog.md` is auto-generated by `generate_changelog.py`. It only shows commits prefixed with `[CONTENT]`. All dev, CI, and formatting commits are silently excluded.

**To appear in the changelog, your commit message must start with `[CONTENT]`:**

```
[CONTENT] Updated DC-GN-001 Section 3.2 MEP file naming for MEP disciplines
[CONTENT] Added LOD 350 requirements table to DC-GN-002 Section 7
```

Commits without this prefix (e.g. `Fix broken image link`, `Update plugin version`) never appear in the public changelog.

## Secrets Required for Deployment

The GitHub Actions workflow requires two repository secrets, configured at:
`Settings → Secrets and variables → Actions`

| Secret | Purpose |
| --- | --- |
| `FTP_USERNAME` | Username for FTP access to `bim.atlstandards.com` |
| `FTP_PASSWORD` | Password for FTP access |

Without these, the Action will fail at the deployment step.
