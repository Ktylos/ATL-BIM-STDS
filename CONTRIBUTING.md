# Contributing to ATL BIM Standards

Thank you for helping improve the ATL BIM Standards. This guide explains how consultants and project teams can file feedback, report errors, and propose amendments.

---

## How Standards Are Maintained

The ATL BIM Standards are maintained by the ATL Planning & Development BIM Team. Standards are updated on a quarterly release cycle and versioned so that every published release remains permanently accessible.

Feedback from consultants is a valued input to this process. The BIM Team reviews all submissions and incorporates accepted changes into the next scheduled release.

---

## Filing Feedback via GitHub Issues

All feedback is submitted through **[GitHub Issues](https://github.com/Ktylos/ATL-BIM-STDS/issues/new/choose)**. Three issue types are available — choose the one that best fits your feedback:

### Standard Clarification

Use this when you need help understanding how a requirement applies to your project. Include:

- Which standard and section you are asking about
- What specifically is unclear
- Your project name (optional, but helpful for context)

The BIM Team will respond with a clarification. If the question reveals genuinely ambiguous language, the response may also trigger an amendment to the published standard.

### Standard Amendment Proposal

Use this when you believe a requirement should be changed, added, or removed. Strong proposals include:

- A clear description of the current requirement
- Exactly what you are proposing instead
- A justification — practical challenges encountered, industry references, or coordination issues

Proposals are reviewed by the BIM Team. Accepted amendments are incorporated into the next release and credited in the changelog.

### Technical Error Report

Use this when you find a factual error, inconsistency between sections, or broken content in the published standards. Include:

- Which standard and section contains the error
- A description of the error (quote the text if possible)
- The correct information, if you know it

Verified errors are treated as high priority and may be patched in an out-of-cycle release.

## What to Expect

| Step | Timeline |
| --- | --- |
| Issue acknowledged | Within 5 business days |
| Initial review / response | Within 10 business days |
| Amendment decision | Next quarterly review cycle |
| Published in a release | On next version tag |

For urgent project-blocking issues, contact the BIM Team directly via the [Support & Contact](https://bim.atlstandards.com/latest/resources/support/) page rather than filing a GitHub Issue.

---

## Accessing Historical Versions

The site maintains two types of published content — a rolling **Latest** version and permanently frozen numbered snapshots. Every page shows a banner identifying which version you are viewing.

If your contract references a specific version of the standards, you can access that exact version at any time:

1. Visit [bim.atlstandards.com](https://bim.atlstandards.com)
2. Use the **version selector dropdown** in the top navigation bar
3. Select the version number referenced in your contract (e.g. `1.0`)
4. Bookmark the resulting URL (e.g. `bim.atlstandards.com/1.0/`) for your project records

All released versions are permanently available and will never be overwritten or removed.
